# current situation

demo/main.js is the main entry point when running `npm run dev`, it is used by developers to test the features of the library, and it is not intended to be used by end-users.
it has the following disadvantages:

- The code in demo/main.js is not optimized for production, and it may contain code that is not relevant to end-users.
- The developers often change the code in demo/main.js, especially in the function "initialConnection" where they change the this.createModel parameters, uncommenting lines. They also need to not forget to comment out the lines they uncommented before committing.
- There is a PWA running in files.html and files.js, but it is not used by developers, and it is intended to be used by end-users. It is usable offline only, does not have a server and does not allow collaboration, and it is not optimized for development. It is also not published to github.io, and it is not intended to be used by end-users.
- demo/main.js adds a lot of menus used by developers, like "open in dashboard", "dark mode", "readonly", "clear", "notify", etc. but also some that are usefull for every use: import/export excel and json.

# ideal situation

the developers run 'npm run dev', it opens a page similar to files.html and files.js, where they can manage their files but

- the files are managed by the same server that runs today using "npm run dev", the commands are sent to the local server and applied the correct document, allowing them to test the multi-user capabilities of the library by default.
- the file manager allows to access similar files than the ones generated by the demo/main.js, but they are generated by main.js (using data.js). All different files are always available like if the user created them all once, but they are created on the fly, keep their ID (so the commands that might have been sent to the server applies to them)
- there is no distinction anymore between demo/index and demo/files: all the features should be available from index, including the file management
- when we publish the library to github.io, we publish the same page as demo/index, but with a different configuration (for example, the server URL is different, and the files are not generated, there is not server handling the files and everything is saved in indexDB and the user can install the PWA to use the application offline

# summary:

merge demo/index and demo/files into a single page that is used for both development and end-users, with different configurations for each use case. The file management should be handled by the same server that runs when using "npm run dev", allowing developers to test the multi-user capabilities of the library by default. The files should be generated on the fly, keeping their ID, and all features should be available from the same page. When publishing to github.io, the same page should be used with a different configuration for end-users and the PWA should be available for offline use.
